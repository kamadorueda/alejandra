# Idempotency Guarantee

## Status
`ready`

## Priority
`P0-critical`

## Context

Formatting the same file twice with Alejandra can produce different output in some cases.
This is a fundamental correctness issue — users and CI pipelines rely on the property that
`alejandra file && alejandra file` produces no changes on the second run. Violations erode
trust and cause CI flakiness.

**Issues**: [#250](https://github.com/kamadorueda/alejandra/issues/250),
[#408](https://github.com/kamadorueda/alejandra/issues/408)

## Scope

**In scope:**
- Add an idempotency check to the test harness (format twice, assert identical output)
- Investigate and fix any non-convergent formatting rules discovered by the check
- Add a `--check-idempotency` CLI flag for users to verify their files

**Out of scope:**
- Performance improvements (this is correctness-focused)
- Style changes (only fix rules that produce non-convergent output)

## Implementation

### Step 1: Add idempotency assertion to test harness

**File**: `src/alejandra/tests/fmt.rs`

The existing test harness calls `format::in_memory()` once and compares to `out.nix`.
Add a second formatting pass and assert convergence:

```rust
// After the existing assertion:
let content_got = alejandra::format::in_memory(
    path_in.to_str().unwrap().to_owned(),
    content_in.clone(),
    config,
).1;

// NEW: Idempotency check — format the output again
let content_got_twice = alejandra::format::in_memory(
    path_in.to_str().unwrap().to_owned(),
    content_got.clone(),
    config,
).1;

assert_eq!(
    content_got, content_got_twice,
    "Idempotency check failed for `{:?}`: formatting twice produces different output",
    case_path
);
```

This goes right after the existing `assert_eq!` that compares against `out.nix`.

### Step 2: Run tests to discover failures

Run `cargo test` and catalog which test cases fail the idempotency check.
Each failure reveals a formatting rule that doesn't converge.

### Step 3: Fix non-convergent rules

For each failing case, investigate the rule responsible and fix it. Common causes:
- Line-width calculations that change after re-indentation
- Comment re-indentation that shifts on each pass
- Whitespace normalization that isn't stable

The specific bugs will depend on what Step 2 reveals. The agent should:
1. Diff the first-pass output vs second-pass output
2. Identify which syntax construct is changing
3. Trace through the relevant rule in `src/alejandra/src/rules/`
4. Fix the rule to produce stable output

### Step 4: Add dedicated idempotency test cases

Create test cases specifically designed to stress idempotency:

**Directory**: `src/alejandra/tests/cases/default/idempotency/`

`in.nix` should contain constructs known to be tricky:
- Deeply nested expressions near the line width boundary
- Comments at various positions (inline, block, between arguments)
- Multiline strings with interpolation
- Mixed indentation in the input
- Long binary operator chains

The `out.nix` will be generated by `UPDATE=1 cargo test`.
